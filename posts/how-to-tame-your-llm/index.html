Let's Start with a List

Taming a large language model doesn't have to be complicated. If you're a busy person, and you're reading this on a subway (like me), or while having lunch, or a coffee, don't worry. Start simple!

LLMs are beautiful machines, and more often than not, simply writing tasks in a markdown checklist and telling the LLM to tick the tasks as they get completed brings attention back to the tasks, which makes it hard for your LLM to forget them.

To try it, paste this prompt in your AI

<INSERT CHECKLIST SKILL PROMPT HERE>

Alternatively, you can also paste this in a markdown file called SKILL.md inside your .claude/skills/checklist folder, and this will be a skill!

Using MCPs Unconventionally

Using a text based list works well, when you have few tasks, and they're not nested. Where it doesn't work well is an example like so:

To understand why something like this might pose a problem, we have to understand the difference between us and LLMs

LLMs have inherently been taught to think in prose. They're able to think better in paragraphs as opposed to clear bullet points. So while the process of hierarchically laying down our tasks might look flawless and appealing to us humans, LLMs will club them together as a paragraph, and focus on the gist, which causes them to tick multiple points at once, and confuse the state of the tasks.

Moreover, if your list is ordered, that increases the chances of hallucination since previous incomplete tasks assumed complete by the LLM will confuse it even further!

One fix to this is making the list "hard" in nature, which means making the process of ticking boxes more explicit. Consider a task checklist like this:

We could explicitly define an MCP server which ... <INSERT PIPELINE DEFINITION HERE/>

Process

Hopefully, I was able to impart useful knowledge about harnesses and how we can go thinking about them through this article. However, we still need a concrete process for actually going ahead and creating a harness.

The best way to start is by just opening your favourite orchestration tool (claude-code for example), walk it through a real usecase for your domain, correct it where its wrong, and record that sample in LESSONS.md

Later, you give points from LESSONS.md to sdlc-skill (where you define intents, constraints, high level design, architectural decision records, and executable implementation specifications for a specific functionality), and build out that component of the pipeline.
