How to tame your LLM

May 16, 2026

After the full intents + SDLC spec back and forth with the LLM, we arrive at a final EIS

But the major challenge is that doesn't guarantee that the LLM is going to reliably follow that spec and deliver solutions

So the LLM is practically REQUIRED to do the following (a Stop hook detects if it checked all the below, if not, LLM is not allowed to exit the loop):

  • Generate test spec (including finite state machines, schema specs, and e2e specs) using sequential thinking type tools that REQUIRE it to think of some key params + reject the tests if they don't heuristically and semantically match our catalog (of around 100+ best practice examples and growing)

  • The LLM is then REQUIRED to launch a maintainer subagent (to spawn dockers, servers and setup dev environment), N x builder subagents (responsible for delivering on the EIS)

  • After they're done, they are REQUIRED to submit screenshots and other proof of work, which are verified DETERMINISTICALLY, HEURISTICALLY, and STOCHASTICALLY by police subagents (which are REQUIRED to submit reason if they pass/fail the builder)

  • A report is submitted to the orchestrator (also called the CEO) agent, which spawns new builder subagents if tests fail

  • After the orchestrator gets everything fixed, it is REQUIRED to commit, push and deploy (using MCP tools ONLY), which DETERMINISTICALLY and HEURISTICALLY verify working.

  • ONLY IF the last tool call is a successful deploy MCP call does the LLM get to exit, else it receives the relevant error, and has to keep trying until it succeeds
